Requirement analysis

The interaction with the client has clarified the meaning of the following terms:


NOUNS:
  • WasteService: a software system named "WasteService", that implements the required automation functions.
  • Service area: a conventional rectangular room, free of internal obstacles and bordered by solid walls. The room must include an Indoor, a PlasticBox and a GlassBox.
  • Indoor: a port used to enter waste material into the service-area.
  • PlasticBox: container, devoted to store objects made of plastic, upto MAXPB kg of material.
  • GlassBox: container, devoted to store objects made of glass, upto MAXGB kg of material.
  • MAXPB: pre-set value representing the maximum capacity of the PlasticBox.
  • MAXGB: pre-set value representing the maximum capacity of the GlassBox.
  • DDR robot: Differential Drive Robot, a robot that works as a transport trolley. It is capable of moving in the environment by executing simple move commands. The transport trolley has the form of a square of side length RD.
  • Home location: fixed starting position inside the service-area where the robot will be located before the software starts and where it will return if no jobs are pending.
  • Deposit action: represents the task of the robot, this activity can be divided into elementary steps:
    • the robot must move from its current position (Home or other) and reach the indoor
    • the robot must pick up the waste load
    • the robot must move from the indoor to the appropriate container (PlasticBox or GlassBox)
    • the robot must deposit the waste in the container.
  • Service-manager: a human which supervises the state of the service-area using the WasteServiceStatusGUI
  • WasteServiceStatusGUI: a user interacting component that allow the service-manager to observe the state of the service-area including:
    • the current state of the transport trolley and it position in the room
    • the current weight of the material stored in the two waste-containers
    • the current state of the Led.
  • Current state of the transport trolley: indicates the status of the trolley, if it is stopped, in motion, or in the HOME position (Idle).
  • Current state of the Led: indicates the status of the led, if it is off, on, or it is blinking.
  • Warning devices: a physical Led that represents the state of the transport trolley.
  • Alarm devices: a physical Sonar that operates as a distance detector, if the measured distance is less than DLIMIT the robot must be stopped, it will be reactivated only when the distance will return to be greater than DLIMIT.
  • DLIMIT: maximum fixed distance below which the robot must be stopped.
  • TruckLoad kg: weight of waste transported by the waste truck.
  • Smart devices: a device that allows the driver of the waste truck to communicate with the Waste Service.
  • Loadaccept: the answer sent by the waste service if the waste load can be accepted.
  • Loadrejected: the answer sent by the waste service if the waste load cannot be accepted.

VERBS:
  • Measures a distance: sonar calculates the distance from the objects in front of it.
  • Pick up a waste-load: the robot takes the load from the waste-truck that goes away and leaves the Indoor free.
  • Deposit the waste-load: the robot unloads the waste in the appropriate container.
  • Must be stopped: the transport trolley must be stopped at its current location, regardless of the action it is taking. This operation is up to the WasteService system.
  • Be resumed: the transport trolley must be reactivated in the same position in which it was stopped and resume the action it was performing before stopping. This operation is up to the WasteService system.

Formalisation of the Requirements

In order to formalize the requirements we will use the QActor meta-model (Qak) developed by our software house. The documentation for this meta-model can be found here : QActor meta-model.

Truck-WasteService interaction: Request-reply

From the requirements, and on the basis of what has been clarified by the client, it is clear that the interaction between Waste Truck and WasteService occurs through the exchange of messages that can be formalized in the following way through the Qak meta-model: 
        
Request storeload : storeload(MATERIAL, WEIGHT)
Reply loadaccept : loadaccept()
Reply loadrejected : loadrejected()
